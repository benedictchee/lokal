# Directory Layout

This monorepo holds every subsystem of the travel marketplace program. The
schema in `proto/` is the single source of truth; every other directory either
generates from it or consumes the generated artifacts.

> Status: **planning**. The tree below is the intended layout. Directories are
> created as each subsystem is brought online, not all at once.

```
travel/
├─ proto/                  # Single source of truth — all .proto schemas + Buf config
│  ├─ buf.yaml             # Module, lint rules, breaking-change rules
│  ├─ buf.gen.yaml         # Codegen targets (TypeScript + Dart)
│  └─ travel/
│     ├─ commerce/v1/      # Catalog, SKU, cart, order, payment messages + services
│     ├─ merchant/v1/      # Merchant account + SKU-management services
│     └─ ai/v1/            # AI chat, embedding, and retrieval messages + services
│
├─ apps/                   # Deployable units (one per subsystem)
│  ├─ consumer-app/        # [1] Flutter + Shorebird mobile client (Dart)
│  ├─ consumer-api/        # [2] CF Worker + D1 — browse, cart, payment, AI chat (Connect server)
│  ├─ merchant-api/        # [3] CF Worker + D1 — merchant account + SKU management (Connect server)
│  ├─ data-pipeline/       # [4] CF Worker + Vectorize + Queues + cron — scrape, embed, index
│  └─ merchant-web/        # [5] Next.js static export — merchant console (served from R2 via Worker)
│
├─ packages/               # Shared, non-deployable libraries
│  ├─ proto-ts/            # Generated TS (protobuf-es + connect-es); consumed by all TS apps
│  ├─ proto-dart/          # Generated Dart (protoc-gen-dart + connect-dart); path dep of consumer-app
│  └─ shared/              # Cross-cutting TS utilities (auth, error mapping, env typing)
│
├─ infra/                  # Operational config — wrangler.jsonc per Worker, D1 migrations, IaC
│
├─ .circleci/              # CI/CD — dynamic config (see CICD.md)
│  ├─ config.yml           # Setup pipeline: maps changed paths → parameters
│  └─ continue-config.yml  # Continuation pipeline: guarded per-subsystem workflows
│
├─ Directory.md            # This file
├─ DataFormat.md           # Protobuf + Connect protocol conventions
├─ CICD.md                 # CircleCI pipeline strategy
├─ Claude.md               # Project memory: setup + baseline requirements
├─ pnpm-workspace.yaml     # pnpm workspace definition (apps/* + packages/*, TS only)
└─ turbo.json              # Task orchestration for the TS workspace (optional)
```

## What lives where

| Path | Purpose | Language / Stack |
|------|---------|------------------|
| `proto/` | Canonical schema for every cross-subsystem message and RPC service. Nothing here is hand-written code — only `.proto` and Buf config. | Protobuf / Buf |
| `apps/consumer-app/` | Consumer mobile app: browse marketplace, purchase, chat with AI. Ships OTA via Shorebird. | Flutter / Dart |
| `apps/consumer-api/` | Consumer-facing edge API: catalog browsing, cart, payment orchestration, AI chat endpoint. | CF Worker + D1 |
| `apps/merchant-api/` | Merchant-facing edge API: account, onboarding, SKU CRUD. | CF Worker + D1 |
| `apps/data-pipeline/` | Background work: scheduled scraping, data prep, embedding, Vectorize indexing. No public surface. | CF Worker + Vectorize + Queues |
| `apps/merchant-web/` | Static merchant console; talks to `merchant-api` over Connect. Built as a static export, stored in R2, served by a Worker. | Next.js (static) + R2 |
| `packages/proto-ts/` | Generated TypeScript stubs and Connect clients/servers. Regenerated by `buf generate`; not edited by hand. | Generated TS |
| `packages/proto-dart/` | Generated Dart messages and Connect client. Regenerated by `buf generate`; not edited by hand. | Generated Dart |
| `packages/shared/` | Code reused across Workers (auth middleware, error→Connect-code mapping, typed env). | TypeScript |
| `infra/` | Deployment and data config kept out of app code: per-Worker `wrangler.jsonc`, D1 schema migrations, optional Terraform/Pulumi. | Config / IaC |

## Conventions

- **Generated code is committed-ignored output**, never hand-edited. Treat
  `packages/proto-ts/` and `packages/proto-dart/` as build artifacts produced by
  `buf generate` from `proto/`.
- **One deployable per `apps/` folder.** A subsystem that needs to be deployed
  independently gets its own folder and its own `wrangler.jsonc`.
- **Shared code is a `package`, never a relative import across apps.** Apps
  depend on `packages/*`; apps never import from each other.
- **TS apps/packages join the pnpm workspace.** The Flutter app stands outside
  the pnpm workspace and is managed by its own `pubspec.yaml`.

See `DataFormat.md` for how these pieces talk to each other.
